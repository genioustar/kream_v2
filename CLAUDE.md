## agnet team으로 작업할때 작업 원칙
- 각 팀원이 변경하기 전에 plan approval을 받아야함.

## 소스수정원칙
- 소스가 추가되거나 수정되면 ARCHITECTURE.md에 변경된 아키텍처를 작성할 것.
- ARCHITECTURE.md가 없으면 프로젝트 구조를 분석하여 작성할 것.

## 하네스 엔지니어링 원칙
코드와 문서가 하나의 사실(source of truth)로 수렴하도록 강제한다. 코드를 바꾸면 같은 커밋 안에서 문서·결정사항·잔재 코드까지 함께 정리한다. 죽은 코드는 읽는 사람과 Claude 모두를 혼란시키므로 남기지 않는다.

### 1. 구현 방식이 바뀌면 잔재를 같은 커밋에서 제거한다
- DOM 파싱 → API 인터셉트, stealth → CDP 같은 **경로 교체**가 일어나면, 이전 경로에서만 쓰이던 함수·상수·헬퍼·import·JS 문자열을 즉시 삭제한다.
- "혹시 나중에 쓸까봐" 남기지 않는다. 필요해지면 git history에서 복원한다.
- 제거 대신 주석 처리(`# TODO`, `# 미사용`)는 금지. 주석은 정보를 숨기고 죽은 코드를 영속화한다.

### 2. 변경 후 호출처 0회면 삭제한다
- 함수·상수를 수정하거나 제거 후보로 식별했으면, `grep -rn "<이름>" --include="*.py"` 로 호출처를 확인한다.
- 정의 위치 외 호출이 0회면 제거 대상이다. 정의도 함께 삭제한다.
- 관련 import(`from X import Y`)도 사용처 소거 시 함께 제거한다 (타입 힌트 전용이면 유지 여부 별도 판단).

### 3. ARCHITECTURE.md와 코드는 1:1 일치해야 한다
- 함수·상수를 추가·삭제·시그니처 변경 시, **동일 커밋 안에서** ARCHITECTURE.md의 해당 모듈 섹션을 갱신한다.
- 문서에는 **현재 호출되고 있는** 함수만 기술한다. 정의만 되고 호출되지 않는 코드는 문서에서 제외 후 제거.
- 디렉토리 트리는 실제 `ls` 결과와 일치해야 한다. 사라진 파일·새 파일 반영 필수.

### 4. 뒤집힌 결정은 .planning/STATE.md 에 명시적으로 SUPERSEDED 처리한다
- `## Decisions` 항목 중 실제 코드가 반대로 구현된 것이 발견되면, 삭제하지 말고 다음 형식으로 덮어쓴다:
  ```
  - [영역] 새 결정 — 사유
    - SUPERSEDED: 기존 결정 내용 — 뒤집힌 이유
  ```
- 이렇게 해야 "왜 이렇게 바뀌었는지"의 컨텍스트가 남는다.

### 5. history.md 의 "제거·교체" 기록은 코드에서도 실제 제거되었는지 같은 날 검증한다
- "A를 B로 교체"로 기록했다면 A가 코드에서 사라졌는지 `grep` 으로 확인.
- 기록만 남고 코드는 그대로인 경우가 가장 위험한 상태. 커밋 전에 반드시 확인.

### 6. 세션 종료 전 정합성 점검 체크리스트
- [ ] 새로 추가·제거한 함수/상수가 ARCHITECTURE.md에 반영되었는가
- [ ] 주석 처리된 코드(`# 구버전`, `# 사용 안함` 등)가 남아있지 않은가
- [ ] history.md 의 최신 항목이 실제 코드 상태와 일치하는가
- [ ] STATE.md 의 Decisions 가 현재 구현 방식과 모순되지 않는가

## 크롤러 검증 시 Playwright MCP 활용 규칙

크롤러 코드를 수정하면 글로벌 CLAUDE.md 의 Evaluator 단계에서 **Playwright MCP**로 대상 사이트의 실제 DOM을 읽어 크롤링 JSON과 교차 검증한다. Kasada·Akamai·Kream SPA 를 우회하려면 실렌더링 브라우저가 필수이므로 Firecrawl·Fetch 같은 정적 HTML MCP 는 사용 금지.

### 설치 (최초 1회, 사용자가 직접 실행)
```
claude mcp add playwright npx @playwright/mcp@latest
```
설치 후 `/mcp` 로 connected 상태 확인.

### 세션 공유 원칙
- MCP 브라우저와 크롤러는 `chrome-debug.sh` 세션(포트 9222)을 **순차 공유**한다 (동시 사용 금지 — 포트 충돌·세션 오염).
- 검증은 크롤러 실행이 **종료된 후** 시작한다.
- 쿠키·fingerprint 오염을 줄이기 위해 **수정된 크롤러만** 대상으로 검증한다 (전체 사이트 일괄 검증 금지).

### 검증 절차 (Evaluator Agent 수행)
1. 크롤러 JSON 로드: `output/YYYYMMDD/<site>_products.json`
2. Playwright MCP 로 대상 URL 오픈:
   - nike: `config.NIKE_SALE_URL`
   - adidas: `config.ADIDAS_SALE_URL`
   - naver: `config.SEARCH_URLS[*].url`
   - kream: 검색 쿼리 기준 페이지
3. DOM 실측값 수집:
   - 총 상품 카드 수
   - 랜덤 샘플 3개의 (상품명, 가격, 모델코드)
   - 스크린샷 저장 → `output/YYYYMMDD/validation/<site>.png`
4. 교차 대조:
   - JSON 상품 수 ≥ 하한선 (nike: 80+, adidas: 300+, naver: 사이트별 상이, kream: 1+)
   - 샘플 3개의 모델코드가 JSON 에 존재
   - 가격 일치 (±1원 반올림 허용)
5. 판정:
   - **PASS** — 모든 항목 충족 → 커밋 진행
   - **FLAG** — 상품 수 하한선 미달이지만 샘플 일치 (세일 종료 가능성) → 사용자 확인 요청
   - **FAIL** — 샘플 불일치 또는 DOM 접근 실패 → Generator 재작업

### 스킵 금지
- 정적 HTML MCP(Fetch, Firecrawl)로 SPA·탐지 사이트를 검증하지 말 것
- 검증 없이 "DOM 이 같을 것"이라 추정하지 말 것 (실측 필수)
- 글로벌 규칙의 Evaluator 호출을 MCP 검증으로 대체하지 말 것 — MCP 검증은 Evaluator 가 사용하는 **도구**이지 Evaluator 역할 자체가 아니다

## MD 파일 컨텍스트 최적화 원칙
대화 세션 종료 전 또는 사용자가 요청 시, 아래 기준으로 md 파일을 정리할 것.

## ARCHITECTURE.md
- 현재 코드 기준으로 정확한 내용만 유지. 삭제된 함수·상수·파일 설명은 제거.
- 디렉토리 구조 트리가 실제 파일 목록과 일치하는지 확인.
- 모듈 설명은 현재 동작하는 public API·핵심 로직만 기술. 내부 구현 세부사항은 코드 주석으로 위임.

## history.md
- 같은 날 동일 기능을 여러 차례 수정한 항목(1차·2차·3차 등)은 최종 결과 하나로 통합.
- 중간 실패 과정·디버깅 시도·롤백 내역은 제거. 최종 결정사항과 그 이유만 남길 것.
- 현재 코드에 이미 반영된 변경사항은 한 줄 요약으로 축약 가능.
- 삭제되거나 대체된 구현에 대한 설명은 제거.
